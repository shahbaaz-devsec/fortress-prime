# FORTRESS PRIME — Architecture & Design

This document describes the internal architecture of the FORTRESS PRIME
hardening tool. It is intended for contributors, auditors, and users who
want to understand how the tool makes decisions before deploying it on
their systems.

---

## 1. High‑Level Design

FORTRESS PRIME is a **single‑file Python application** with zero external
dependencies beyond the Python 3.10+ standard library. It runs as `root`
because it modifies system‑level configuration files, kernel parameters,
firewall rules, and package state.

The tool operates in three distinct modes:

| Mode | Flag | What It Does |
|------|------|--------------|
| **Apply** | *(default)* | Executes every enabled hardening step, then verifies the result. |
| **Dry‑run** | `--dry-run` | Prints what *would* be changed without modifying the system. |
| **Verify** | `--verify` | Runs only the verifier functions — reports current posture without any changes. |

Additionally, a `--rollback` flag executes the most recent `rollback.sh` file
generated by a previous run.

## 2. Component Overview

```
                     ┌────────────────────────────┐
                     │         main()             │
                     │  Parse args → Hardener.run │
                     └────────────┬───────────────┘
                                  │
                    ┌─────────────▼──────────────┐
                    │        Hardener            │
                    │  ┌───────────────────────┐ │
                    │  │   StepRegistry        │ │
                    │  │   (55 steps)          │ │
                    │  └───────────────────────┘ │
                    │  ┌───────────────────────┐ │
                    │  │   FileManager         │ │
                    │  │   (backup, rollback)  │ │
                    │  └───────────────────────┘ │
                    │  ┌───────────────────────┐ │
                    │  │   Verifier Functions  │ │
                    │  └───────────────────────┘ │
                    └────────────────────────────┘
```

### 2.1 `Hardener` Class

The central orchestrator. It holds all hardening logic, the step registry,
and the file manager. It receives configuration (admin user, SSH port, etc.)
from the CLI argument parser.

### 2.2 `StepRegistry`

A simple list of dictionaries. Each entry maps a step ID (e.g., `"02"`)
to:

- A human‑readable name
- CIS and NIST control references
- A callable `func` (the hardening logic)
- A callable `verify` (the post‑run verifier)

This design makes adding new steps trivial — register a function, write its
verifier, and add a tuple. No other code needs to change.

### 2.3 `FileManager`

Every file the script touches is tracked in a **rollback ledger**. Before
modification, the original file is backed up to
`/var/backups/fortress-prime/<run-id>/`. Atomic writes are used (write to
temp file → `os.replace`) to prevent partial writes. After a run, a
`rollback.sh` script is generated that can restore the file contents.

**Limitation:** The rollback restores file contents only. It does **not**
undo package installs, systemd unit state changes, UFW rule additions, or
sysctl runtime changes. See `KNOWN_LIMITATIONS.md`.

## 3. Execution Flow

1. **Pre‑flight**: Verify we are on Ubuntu 24.04; ensure we are root.
2. **Idempotency check**: For each step, the verifier runs *before* the
   step. If the verifier returns `pass`, the step is skipped entirely.
3. **Apply**: If the step is not already applied, `func()` is called.
4. **Verify**: The verifier runs again. If it fails (`fail`), the script
   retries the step **once**. If it still fails, the error is logged and
   the step is marked `failed` — the tool continues with the next step.
5. **Critical‑step abort**: Steps `02` (SSH) and `03` (Firewall) are
   designated **critical**. If either fails during apply, the script
   aborts immediately to prevent lockout.
6. **Report**: A `RunReport` dataclass collects the result of every step
   and is serialised to a JSON file under `/var/log/fortress-prime/`.
7. **Rollback script**: The `FileManager` writes `rollback.sh` to
   `/var/backups/fortress-prime/`.

## 4. Safety Guarantees

| Scenario | Protection |
|----------|------------|
| SSH config syntax error | `sshd -t` runs **before** the service is reloaded. On failure, the drop‑in config is removed. |
| Firewall lockout | The **current** SSH client’s source IP is automatically added as a temporary UFW allow rule before the default‑deny policy is applied. |
| Sudoers syntax error | `visudo -cf` validates every sudoers drop‑in. On failure, the file is removed. |
| Root account locked without a key | Step 20 checks that the admin user exists and has a **non‑empty** `authorized_keys` file before locking `root`. |
| Post‑run SSH reachability | Step 29 performs a TCP connect to `127.0.0.1:<ssh-port>` after all changes. Failure is logged explicitly. |

## 5. Configuration

All configuration is passed via **command‑line flags**. There is no config
file — this is intentional to keep the tool self‑contained and auditable.

| Flag | Default | Purpose |
|------|---------|---------|
| `--admin-user` | `deploy` | The non‑root user to create/ensure |
| `--ssh-port` | `22` | The SSH port to enforce |
| `--allow-from` | *(empty)* | Comma‑separated CIDRs allowed to reach SSH |
| `--hostname` | *(empty)* | Hostname to set |
| `--enable-auto-updates` | `False` | Start unattended‑upgrades immediately |
| `--disable-ipv6` | `False` | Opt‑in IPv6 disable |
| `--enable-usbguard` | `False` | Opt‑in USBGuard installation |
| `--enable-clamav` | `False` | Opt‑in ClamAV installation |
| `--enable-haveged` | `False` | Opt‑in haveged installation |
| `--dry-run` | `False` | Preview only |
| `--non-interactive` | `False` | Skip confirmation prompt |
| `--only` | *(empty)* | Comma‑separated step IDs to run |
| `--skip` | *(empty)* | Comma‑separated step IDs to skip |
| `--verify` | `False` | Run verifiers only |
| `--rollback` | `False` | Execute the most recent rollback.sh |
| `--report-dir` | `/var/log/fortress-prime` | Where to write the JSON report |
| `--verbose` | `False` | Enable DEBUG‑level logging |

---

## 6. Logging & Observability

- **Console**: INFO‑level, colourised (auto‑disabled if stdout is not a TTY).
- **File**: Rotating file handler at `/var/log/fortress-prime/fortress-prime.log`
  (max 10 MiB per file, 10 backups).
- **Syslog**: Best‑effort forwarding to `/dev/log` with facility `LOG_AUTH`.
- **Audit JSON**: Every run produces a timestamped `audit_report_<run-id>.json`
  with per‑step status, duration, verification result, and error details.

---

## 7. Compliance Mapping

Every step carries a `cis` and `nist` field in the registry. These are used
only for reporting — they do not affect execution. The mapping is a best‑effort
alignment, not a formal compliance certification. See `KNOWN_LIMITATIONS.md`
for the honest assessment of coverage.

---

## 8. Extending the Tool

To add a new hardening step:

1. Write a method `step_<your_step>()` that returns a `(status, detail)` tuple.
2. Write a method `verify_<your_step>()` that returns a `(verify_status, verify_detail)` tuple.
3. Register both in `_register_steps()` using `self.registry.add(...)`.
4. If the step requires a new CLI flag, add it to `parse_args()` and pass it
   through to the `Hardener` constructor.

The step will automatically participate in idempotency checks, verification,
retry‑on‑failure, and audit reporting.

---

## 9. Known Design Trade‑offs

- **Single‑file** vs. modular package: Chosen for zero‑dependency distribution
  and ease of auditing. A 2,800‑line file is large, but the linear structure
  makes it easy to review.
- **CLI flags** vs. config file: Flags keep the tool self‑contained. A config
  file could be added in the future for complex deployments.
- **No parallel execution**: Steps run sequentially. This avoids race conditions
  and makes the output predictable, but extends total run time (especially
  AIDE initialisation, which can take 20+ minutes).
- **File‑based rollback only**: Full system rollback (packages, services) is
  not implemented. A VM snapshot is recommended before running.
